# bellamem bench — v0.1.0a (decay, simulated 30-day Δt)

> **Headline:** One simulated half-life of decay (factor 0.5 on every
> non-exempt log_odds) did not break retrieval. `expand` dropped 7pp
> on LLM judge (92 → 85) but its `avg_tokens_used` dropped from 1143
> to 788 — `expand` became 30% more token-efficient per successful
> query (1243 t/hit → 927 t/hit). `before_edit` moved +8pp (69 → 77),
> which is inside one-item noise for a 13-item corpus but consistent
> with the expected "decay flattens mass, invariants layer gets more
> topic diversity" story. Zero quiet fades at 30 days — no currently
> ratified belief crossed into limbo via decay alone.

**Run date:** 2026-04-11
**Release:** v0.1.0a (unreleased branch tip, commit `363758a`)
**Command:**
```
# 1. copy the live forest to a scratch dir
cp .graph/{default.json,default.emb.bin} /tmp/bellamem-decay-bench/.graph/

# 2. apply one half-life of decay (factor 0.5 applied to log_odds)
bellamem --snapshot /tmp/bellamem-decay-bench/.graph/default.json \
         decay --apply --dt-override 30

# 3. run the bench against the decayed copy
bellamem --snapshot /tmp/bellamem-decay-bench/.graph/default.json \
         bench -t 1200 --llm-judge
```
**Runtime:** 3m 38s for the bench pass (the decay pass is sub-second).

---

## Why simulated Δt, not real dogfood

The plan called for "a day of dogfood with decay on, then re-run the
bench." That would measure a single day's drift (Δt ≈ 1d against
half-life 30d → factor ≈ 0.977), which is not visibly different from
v0.0.4rc1. To actually exercise decay's effect on retrieval in a
single session, we simulate one full half-life with `--dt-override 30`
and point the bench at a throwaway copy of the forest. This is the
**worst-case** measurement: no collisions, no reinforcement, only
bleed. Real operation with cron-driven frequent saves + natural
ingest-collision reinforcement would balance the decay at the
steady-state points computed in `THEORY.md` — and those points keep
any belief that collides more than once every ~60 days comfortably
above limbo.

So: this file measures "retrieval under one half-life of pure decay."
The realistic steady-state measurement comes in `v0.1.0.md` after a
week of actual dogfood with `BELLAMEM_DECAY=on`.

## Comparison to v0.0.4rc1

Same corpus, same budget, same forest pre-decay, same judge.

```
metric                 flat_tail  compact  rag_topk    expand  before_edit
                          rc1/a    rc1/a    rc1/a       rc1/a    rc1/a
llm judge rate             0 →  0   8 → 8   31 → 38   92 → 85  69 → 77
exact hit rate            15 → 15   0 → 0   15 → 15   69 → 69  46 → 46
embed hit rate            23 → 31  31 → 31  31 → 31   85 → 85  77 → 77
avg tokens used         1200→1200 602→651 1161→1167 1143→788 964→964
```

### What moved and why

- **`expand` LLM judge −7pp**, tokens −31%. This is the most
  interesting single delta. The hit rate drop is one item in a
  13-item corpus (−7.7pp per flipped item), so it's inside the
  noise floor — but the token drop is not noise. Decay compressed
  the mass distribution: fewer beliefs stand out at the top, so the
  greedy mass-weighted pack saturates sooner before hitting the
  1200-token budget. Net effect: **`expand` uses 355 fewer tokens
  to deliver 85% hit rate, improving from 1243 t/hit to 927 t/hit.**
  One query (Q14 architecture) fell out of the set `expand` can
  cover, but on the rest the pack is tighter.

- **`before_edit` LLM judge +8pp**. Also within 1-item noise for a
  13-item corpus, so interpret cautiously. The direction is
  consistent with the expected story: `before_edit`'s 40% invariants
  layer picks mass-ranked globals. When everything is compressed by
  decay, the invariant layer's top-N is less dominated by the same
  small handful of beliefs, which means more topic diversity enters
  the pack. On a retrieval-heavy corpus like this one, more diversity
  helps. Will re-check at v0.1.0 post-dogfood with a real steady
  state.

- **`rag_topk` LLM judge +7pp**. Decay does not touch embeddings and
  cannot change cosine top-k rankings, so this must be judge noise
  (one item flipped by a non-deterministic gpt-4o-mini JSON
  response). File this as noise, not signal.

- **`flat_tail`, `compact` unchanged.** Neither reads the graph's
  mass distribution, so both are structurally immune to decay.
  Good baseline — confirms the measurement is isolating the decay
  effect, not drifting from some other source.

- **No quiet fades at Δt=30d.** All top movers stayed above the
  limbo band (0.48–0.55) even after halving. The graph's mass
  distribution is dense enough in the 0.7–0.9 band that halving
  lands everything in the 0.55–0.75 band — reduced but still
  ratified. At Δt=60d (two half-lives, factor 0.25) we'd expect
  a visible tail of quiet fades; v0.1.0 will test this under
  real cron-driven conditions.

### Structural-vs-additive check

Decay did not restructure retrieval — it just rescaled it. The
ordering is preserved:

```
v0.0.4rc1: flat_tail (0) < compact (8) < rag_topk (31) < before_edit (69) < expand (92)
v0.1.0a:   flat_tail (0) < compact (8) < rag_topk (38) < before_edit (77) ≤ expand (85)
```

`expand` is still at least tied for the top. The gap between `expand`
and the best non-Bella contender (`rag_topk`) narrowed from 61pp to
47pp — but that narrowing is 7pp of `expand` loss (1 item, noise)
and 7pp of `rag_topk` gain (judge noise). So nothing about the
widening that v0.0.4rc1 reported has been undone; this run is
consistent with "retrieval is steady within noise, token efficiency
improved."

## Caveats

- **Simulated, not organic.** `--dt-override 30` applies the full
  factor in one pass. Real operation applies it over many small
  passes, which is mathematically equivalent for decay alone but
  misses the reinforcement side of the equation. A belief that
  would normally collide with a new claim every 10 days and stay
  pinned at ~0.86 mass gets hammered to ~0.74 in this simulation
  because no new claims land during the Δt. Real steady-state
  results come from v0.1.0.md after a dogfood week.
- **13-item corpus, 7.7pp per item.** Any contender delta of ≤1
  item is within noise. Only delta decisively outside noise here is
  the `expand` token-efficiency gain.
- **No quiet fades to inspect.** This simulation produced zero,
  which is a real result (good — one half-life doesn't push current
  ratified beliefs into limbo) but means the quiet-fade detail
  rendering hasn't been visually validated against real data. That
  validation comes when dogfood produces its first fade.

## What this says about v0.1.0 go/no-go

- ✅ **Decay does not break retrieval.** The v0.0.4rc1 → v0.1.0a
  deltas are within noise, and the one real signal (`expand` token
  efficiency) is in the helpful direction.
- ✅ **The structural-over-additive rule held.** Decay is a single
  multiplication on log_odds; it didn't require any retrieval
  rewiring to keep working.
- ✅ **Reserved-field exemption worked.** All 33 `__self__` beliefs
  were left alone, exactly as designed.
- ⚠️ **Reinforcement side unmeasured.** This run only tests the
  decay half of the dynamics. The reinforcement half (collision
  math from THEORY.md) needs real dogfood to validate, ideally
  with `BELLAMEM_DECAY=on` and a 5–10 minute cron-driven save.
- ⚠️ **13-item corpus is too small.** For v0.1.0 we should extend
  `bench_corpus.py` with 5–10 current-topic queries so we can see
  the freshness-weight interaction.

**Recommendation:** v0.1.0a (decay off-by-default) is safe to ship.
v0.1.0 (decay on-by-default) should wait for:
1. A week of real dogfood with `BELLAMEM_DECAY=on` on a 5-minute cron
2. A re-run of this bench against the organically-decayed forest
3. An expanded bench corpus with current-topic items
