# bellamem bench — v0.0.4rc1

> **Headline (budget = 1200 tokens, LLM judge):** `expand` hits 92%,
> `before_edit` hits 69%, everything else ≤31%. As the forest grew
> from the v0.0.2 corpus-authoring state to 1834 beliefs, the gap
> between `expand` and the next-best contender (`rag_topk`) widened
> from **15pp (100 − 85) to 61pp (92 − 31)**. Structured
> mass-weighted retrieval scales with forest size; cosine top-k
> degrades because a larger forest means more noise hits.

**Run date:** 2026-04-10
**Release:** v0.0.4rc1 (commit `f9abe21` + docs commit `d94713a`)
**Command:** `bellamem bench -t 1200 --llm-judge`
**Runtime:** 4m 5s (wall clock, mostly LLM judge round-trips)

---

## Configuration

| | |
|---|---|
| forest | 1834 beliefs, 3 fields, 60 roots |
| embedder | `openai-text-embedding-3-small` (dim 1536) |
| EW | `hybrid` (regex + gpt-4o-mini) |
| judge model | `gpt-4o-mini` |
| tokenizer | `tiktoken` / `cl100k_base` |
| corpus | `bench_corpus.py` — 13 items, unchanged since v0.0.2 |

The forest has grown substantially since v0.0.2 (was a much smaller
dogfood session; now 1834 beliefs across `__self__`,
`ingest_claim_processing`, and `user_invariants_memory`). The corpus
itself is unchanged — same 13 queries, same `expected_any_of`
strings — so this is a clean apples-to-apples measurement of how
each contender responds to forest growth.

## Comparison to v0.0.2

Same budget (1200 tokens), same corpus (13 items), same judge
(`gpt-4o-mini`). Different forest size (small dogfood session →
1834 beliefs).

```
metric                 flat_tail  compact  rag_topk    expand  before_edit
                          v2/v4    v2/v4    v2/v4       v2/v4    v2/v4
llm judge rate            15 →  0   8 → 8   85 → 31   100 → 92  77 → 69
exact hit rate            15 → 15  15 → 0   77 → 15    69 → 69  46 → 46
embed hit rate            23 → 23  38 → 31  85 → 31    85 → 85  69 → 77
```

### What moved and why

- **`expand` is essentially unchanged** (100 → 92 judge, 69 → 69
  exact). The 8pp judge dip is 1 item in a 13-item corpus — within
  noise. The mass-weighted retrieval continues to surface the
  decisive fact even as the forest grows 10×+.

- **`before_edit` is essentially unchanged** (77 → 69 judge, 46 →
  46 exact). Still below `expand` on this retrieval-heavy corpus,
  consistent with the v0.0.2 finding that the 5-layer split is
  tuned for pre-edit bandaid prevention, not general recall.

- **`rag_topk` collapsed** (85 → 31 judge, 77 → 15 exact).
  This is the interesting result. Top-k cosine retrieval on a
  larger forest pulls up more plausible-looking-but-wrong neighbors,
  and the decisive fact gets crowded out of the top-k window. The
  structural advantage of `expand` (mass weighting + field-aware
  ranking + dispute surfacing) pays off exactly when the forest is
  large enough to need it.

- **`flat_tail` hit 0% on the LLM judge** (15 → 0). This is
  *cleaner*, not worse. v0.0.2 documented a self-reference lift at
  ≥3000 tokens because the corpus was drafted in the same session
  that built the tree. With time and ingestion, the corpus-authoring
  moment has moved well out of the flat tail, so the old 15% was
  artifact, not signal. Pure recency genuinely has no idea about
  these queries.

- **`compact` is flat** (8 → 8 judge). The LLM-summary contender is
  independent of forest size by construction — it works from the
  recent transcript window, not the graph. Its performance doesn't
  change as the graph grows.

### The ordering widened

At v0.0.2, the best non-Bella contender (`rag_topk`) was 15
percentage points behind `expand`. At v0.0.4rc1, it's 61pp behind:

```
v0.0.2:  flat_tail (15) << compact (8)  <  before_edit (77) < rag_topk (85) < expand (100)
                                                                          ← 15pp gap

v0.0.4:  flat_tail ( 0)  < compact (8)  <  rag_topk (31) < before_edit (69) < expand (92)
                                                                          ← 23pp gap
                                                          ← 61pp gap to rag_topk
```

**The headline story this file tells isn't "expand dropped 8pp."
It's "rag_topk dropped 54pp while expand held."** That's the
load-bearing evidence for the mass-weighted-retrieval thesis: the
larger the forest, the more the structure matters.

## What's changed in code since v0.0.2

None of these touch retrieval:

- **v0.0.3rc1/rc2** — per-project `.graph/`, automatic R3
  consolidation on ingest, source grounding + narrative replay,
  structural pruning (`bellamem prune`), default `save` → current
  session only with incremental ingest, embed-cache prune bounded
  to live beliefs.
- **v0.0.4rc1** — v3 split storage format (graph JSON + binary
  embeddings sidecar), `bellamem-guard` PreToolUse hook, embedder
  batching in `ingest_session`.

The retrieval code (`core/expand.py`, `core/bella.py` routing, the
Jaynes accumulation, the entity index) is untouched. So the deltas
above are explained entirely by corpus/forest-size effects, not by
algorithm changes.

## Caveats

- **Single budget point.** Only 1200 tokens was measured. A full
  sweep (500 / 1000 / 2000 / 4000 / 10000) would cost ~$0.05 and
  take ~20 minutes. Do that before any release that touches
  retrieval, not for every dogfood measurement.
- **13-item corpus.** One item = 7.7pp. An 8pp delta is within
  noise; a 54pp delta (rag_topk) is decisively outside noise.
- **`expected_any_of` strings are frozen at v0.0.2.** Some may
  reference wording that has since been merged, renamed, or pruned.
  This penalizes `exact` and `embed` hit rates across all
  contenders uniformly — the LLM judge metric is the more robust
  comparison because it accepts paraphrase.
- **Self-reference is reduced, not eliminated.** The v0.0.2
  corpus-authoring turns have moved out of the flat tail (hence
  flat_tail → 0), but later sessions have likely re-mentioned some
  corpus themes, so the graph does still contain some traces of
  the bench content. This inflates all graph-reading contenders
  uniformly.
- **No before_edit-specific corpus.** The corpus is retrieval-heavy;
  `before_edit` is designed for bandaid-prevention at the moment
  of an edit. The 69% vs 92% judge gap is expected by design, not a
  regression — v0.0.2 already documented this at 77% vs 100%.

## Next bench should cover

When v0.1.0 ships log-odds decay:

1. **Re-run at 1200 tokens** to measure the direct effect of decay
   on mass-weighted retrieval — decay will re-rank older beliefs
   downward, which should help `expand` when queries are
   current-topic and may hurt it when queries reach far back.
2. **Multi-budget sweep** so we can see whether decay changes the
   saturation point (v0.0.2 saturated at 800 tokens).
3. **Add 3-5 current-topic queries** to `bench_corpus.py` that
   specifically test freshness-weighted retrieval vs. plain mass.
   The current corpus is mostly decided-decision queries where
   mass dominates — freshness effects are invisible to it.
