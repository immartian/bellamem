# Bellamem v0.3 — Node/TypeScript Rewrite Plan

Decision made 2026-04-15 after ten days of Python v0.2 migration work.
This document is the handoff spec for the port. A fresh Claude session
should read this plus `/bellamem resume` and have enough context to
pick up Phase 0 without re-litigating the decision.

---

## Why rewrite

Bellamem's target audience is **AI-agent builders broadly**, not
Python-first researchers. That audience lives in Node:

- Claude Code itself is Node
- `npx` is the install friction floor; `pipx install` is a tax per user
- Competing memory tools (claude-mem, mem0, langchain memory) ship as
  npm packages — Bella being Python makes it an ecosystem outsider
- Localhost web UIs are frictionless in Node, always one step off-keel
  in Python
- The 85% of the product that determines adoption is *integration*,
  not algorithm — and integration is where language choice matters

Python was the right first-implementation language for research
velocity: NumPy embedding math, mature OpenAI SDK, cleaner schema
ergonomics. That's still true of the *core*. But the core is ~15% of
what ships. The other 85% is friction, and friction is in Node.

## Invariants (non-negotiable across the port)

These are the things that must NOT change during the rewrite. If any
of them change, the port loses its dogfood signal and users lose their
graph.

### 1. Byte-compatible `.graph/v02.json`

The TS port reads and writes the exact same JSON schema as the Python
v0.2-ref. Users upgrade transparently. Python and TS can run side-by-
side against the same graph file during development — that's the diff
harness (see Phase 1).

Specifically: `Source`, `Concept`, `Edge` must serialize to the same
fields with the same types in the same order. The `concepts` key stays
a dict keyed by id, `edges` stays a list, `sources` stays a dict.
Field names, enum values, and mass formulas are frozen.

### 2. v0.2 schema is frozen during the port

- `class`: invariant / decision / observation / ephemeral
- `nature`: factual / normative / metaphysical
- `state` (ephemeral only): open / consumed / retracted / stale
- `edge types`: support / dispute / cause / elaborate / voice-cross /
  retract / consume-success / consume-failure
- R1 mass accumulation via log-odds (`_logit`/`_sigmoid`,
  `_MASS_DELTA_NEW_VOICE=0.5`, `_MASS_DELTA_REPEAT_VOICE=0.1`)
- `slugify_topic` produces identical ids for identical topic strings

Any schema evolution waits for v0.4. No exceptions.

### 3. Port only the v0.2 slash-command surface

In scope:
- `/bellamem resume` — `proto.resume_text`
- `/bellamem save` — `proto.ingest_session` + the per-project flock
- `/bellamem recall | why | ask` — `proto.walker.ask_text`
- `/bellamem audit` — `proto.audit.audit` + `format_audit`
- `/bellamem replay` — `proto.replay.replay_text`
- `bellamem-guard` PreToolUse hook
- Dogfood cron (scripts/bellamem-dogfood-cron.sh)

Out of scope (dead weight, do not port):
- `bellamem/core/` — the flat-graph `Bella` object, `bella.py`,
  `ops.py`, `gene.py`, `expand.py`, `replay.py`, `prune.py`,
  `decay.py`, `scrub.py`, `surprise.py`, `visualize.py`, `tokens.py`
- `bellamem/adapters/chat.py` — used only by old scenarios tests
- `bellamem/adapters/llm_ew.py` — flat-graph EW extractor
- `bellamem/bench.py`, `bench_corpus.py` — flat-graph benchmark harness
- `cmd_expand`, `cmd_before_edit`, `cmd_emerge`, `cmd_prune`,
  `cmd_decay`, `cmd_scrub`, `cmd_surprises`, `cmd_entities`,
  `cmd_render`, `cmd_bench`, `cmd_ingest_cc`
- Tests for any of the above

The port is the excuse to delete all of this for good.

### 4. Dogfood continuity

Day-one TS port must ingest the current bellamem repo's session
history with zero data loss. This is how we know extraction quality
hasn't regressed. If the Node version reads `.graph/v02.json` and
runs a `save --tail 50` against today's live session and comes out
with the same class×nature distribution and similar cost, we trust it.

## Phase 0 — Stabilize Python v0.2-ref (1 week)

The point of Phase 0 is to **freeze a diff target**. Rewriting a
moving target doubles work. Phase 0 produces three artifacts:

### 0.1 Write the v0.2 spec document

`docs/rewrite/v0.2-spec.md` — language-independent description of what
v0.2 IS. JSON Schema blocks for Source/Concept/Edge. Prose for the
mass accumulation formula, the edge-type semantics, the ephemeral
state machine, the ingest loop, the classifier prompt contract. This
is what the TS port implements against.

Source material:
- `bellamem/proto/schema.py` — the type definitions
- `bellamem/proto/graph.py` — the container + indices + dedup
- `bellamem/proto/ingest.py` — the ingest loop and R1 wiring
- `bellamem/proto/resume.py` — the epistemic section layout
- `bellamem/proto/walker.py` — the retrieval semantics
- `bellamem/proto/audit.py` — the 5 health signals + thresholds
- `bellamem/proto/replay.py` — the chronological view
- `memory/project_graph_v02_schema.md` — the original design memo

The spec document should be readable by someone who has never seen
the Python code. ~2000 words, heavy on examples.

### 0.2 Cut the v0.2.0-ref tag

After the spec is written and any last critical fixes land, tag
`v0.2.0-ref` on master. That commit is the reference implementation.
Python work freezes here. No more Python features, bug fixes, or
refactors after this point except as diff-harness tooling during
Phase 1.

### 0.3 What to fix before freezing, what to defer

**Fix now (cheap in Python, painful to rediscover in TS):**
- Nothing urgent. Today's ten commits already closed the major gaps
  (walker, audit, replay, streaming reader, per-project lock, path
  encoding, platformdirs).

**Defer to TS port (prompt/classifier issues, fix once across both):**
- Retract-wiring false positives (turn-level cue words like "stop",
  "wait", "now" mis-attributed as retractions). This is a classifier
  prompt bug — fix it in the TS port's prompt rather than twice.
- Metaphysical class under-population in the extractor. Same shape:
  prompt issue, not language issue.
- Over-eager concept minting (surface-word → concept without
  referent grounding). Walker-primitive work, cross-cuts the port.

**Park (not in Phase 0 scope at all):**
- The 55+ open ephemerals in the current graph. Most are stale or
  superseded by today's work. Let the port fresh-eye what's still open.

## Phase 1 — TS port of v0.2-ref core (2–3 weeks)

Module layout:

```
packages/bellamem/
  src/
    schema.ts        — Source, Concept, Edge interfaces + slugify (~150 LoC)
    graph.ts         — Graph container, indices, nearest_concepts,
                       find_similar_concept, R1 cite() (~350 LoC)
    store.ts         — load_graph / save_graph / atomic write (~80 LoC)
    walker.ts        — ask_text, relevance + 1-hop edge walk (~300 LoC)
    audit.ts         — 5 signals + format_audit (~200 LoC)
    resume.ts        — resume_text, typed section layout (~250 LoC)
    replay.ts        — replay_text, session picker by timestamp (~150 LoC)
    ingest.ts        — read_session_turns (streaming) + ingest_session
                       loop + R1 + R5 sweep (~500 LoC)
    clients.ts       — Embedder + TurnClassifier, OpenAI SDK,
                       sha256-keyed disk caches (~250 LoC)
    env.ts           — XDG user config .env loader, project .env,
                       shell env precedence (~80 LoC)
    paths.ts         — project_root, graph path resolution, Claude
                       Code project_dir_for (~100 LoC)
    cli.ts           — commander-based command dispatch (~400 LoC)
    guard.ts         — PreToolUse hook body, stdin JSON protocol,
                       v02.json reader, advisory pack builder (~250 LoC)
  bin/
    bellamem.ts      — CLI entry point
    bellamem-guard.ts — guard hook entry point
  test/
    *.test.ts        — vitest, port 189 tests from Python (~1500 LoC)
```

Total target: **~4,000 lines of TS** from the ~15,000-line Python
codebase. The compression is real because `core/` and friends are
dropped entirely.

### Diff harness

Before declaring the port done, build `scripts/diff-python-vs-ts.sh`
that runs the same command against the same `.graph/v02.json` with
both Python and TS, then diffs the outputs. Used for every major
subcommand (resume, recall, ask, why, audit, replay). Any drift is
a bug in the port.

### Tests first, then implementation

Port the vitest tests before (or alongside) the implementation for
each module. The Python tests are the spec in code form — they've
been debugged over a year. Don't lose that signal by reimplementing
behavior from prose.

## Phase 2 — Migration (1 week)

### 2.1 npm package structure

```
@bellamem/core   — the TS library (Graph, walker, audit, etc.)
@bellamem/cli    — the CLI, depends on @bellamem/core
@bellamem/guard  — the PreToolUse hook binary
```

Installable via `npx @bellamem/cli install`, which:
1. Installs itself globally if not already
2. Writes `~/.claude/commands/bellamem.md` pointing at the Node binary
3. Writes `~/.config/bellamem/.env` template (XDG user config)
4. Prints a "set OPENAI_API_KEY, then try /bellamem save" onboarding

### 2.2 Claude Code slash command

Update `~/.claude/commands/bellamem.md` to run `bellamem` (the Node
binary) instead of the Python `bellamem`. If `bellamem` resolves to
both Python and Node installs, the Node one wins (user should
`pipx uninstall bellamem` after confirming the Node version works).

### 2.3 Dogfood cron

Replace `scripts/bellamem-dogfood-cron.sh` with a Node-native timer
using `setInterval` OR a systemd user timer invoking the Node CLI.
Systemd is cleaner and matches the per-project lock path.

### 2.4 Python v0.2-ref archive

Python version stays on GitHub as `v0.2.0-ref` tag. README updated
to point new users at the Node version. No new Python releases.
The Python `bellamem/` directory can be moved to `python-ref/` or
deleted entirely once the Node version has a month of dogfood and
the v0.2 graph reads correctly from both.

## Phase 3 — Native Node wins (ongoing)

Things that were always going to be painful in Python and are
trivial in Node:

- `npx @bellamem/cli install` — the native install story
- Localhost web UI — Fastify or Hono serving the existing D3/Cytoscape
  templates, live graph data, recent recalls, mass distribution, etc.
- Tighter Claude Code integration — same runtime, shared transcript
  parser, potentially inline with Claude Code's own module system
- `bellamem-guard` as a long-running daemon instead of a shell-out
  per tool call (Claude Code hook protocol supports this)
- Progressive walker mode (search → walk → expand) as a natural
  extension of walker.ts

## Realistic timeline

- Phase 0: 1 week (spec + freeze)
- Phase 1: 2–3 weeks (TS port with diff harness)
- Phase 2: 1 week (migration)
- Phase 3: ongoing

**Calendar estimate: 6–8 weeks to v0.3.0-alpha.** Dogfooding takes the
longest — you need at least a week of the Node version eating its own
transcripts before trusting the extraction.

## Risks to name explicitly

1. **The retract-wiring bug ports cleanly as a prompt bug.** Good —
   you fix it once. But don't assume a rewrite magically improves
   extraction quality. That needs prompt engineering, not language
   change.

2. **Schema drift risk.** If anything about `.graph/v02.json` changes
   during the port, the Python guard breaks for old users and the
   dogfood data becomes unreadable. The TS port must refuse schema
   changes until after v0.3.0 ships.

3. **Test parity is the quiet killer.** Porting 189 tests isn't hard.
   Verifying that Node gives the *same* answers as Python for the
   same inputs is where most rewrites silently diverge. Budget real
   time for the diff harness; don't skip it.

4. **Dogfood continuity.** The current 494+-concept v0.2 graph is the
   most valuable artifact on disk. The Node port must read it
   unchanged from day one so dogfood signal survives.

5. **Research claims re-anchor.** The $1.20 cost footprint, the 95%
   single-voice rate, the LLM-judge bench results — these are
   Python-era numbers. Re-measure in the Node version before making
   public claims.

6. **Opportunity cost during the 6–8 week window.** No new user-
   visible features while the port is underway. If Bella has active
   external users, that's friction. If Bella is still pre-public,
   it's fine.

## What fresh-me should do on resume

1. `/bellamem resume` — verify the graph carries forward cleanly.
2. Read `memory/project_v03_node_rewrite.md` — the memory note.
3. Read this document (`docs/rewrite/v0.3-node-plan.md`).

## Not decided

- **Exact Node module format.** ESM vs CJS. Probably ESM since it's
  2026 and TS 5+ handles this fine.
- **Test runner.** Vitest is the assumption but Node's built-in
  `node:test` is also viable and dependency-free.
- **Embedding library.** `@openai/openai` for API calls, but no
  equivalent of NumPy for vector math. Either inline cosine (simple)
  or use `ml-matrix` / `vector` / `compromise`. Inline is fine for
  v0.3.0 — dimensions are 1536 for `text-embedding-3-small`, cosine
  is one loop.
- **Whether to keep `python-ref/` in the same repo or split it out.**
  Probably keep for a few months — users porting from Python need to
  see the reference.
- **Whether to rename the project.** `bellamem` → `@bellamem/core`
  is the minimum change. The word "bellamem" is fine and searchable.
