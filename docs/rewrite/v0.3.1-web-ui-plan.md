# Bellamem v0.3.1 — Localhost Web UI Plan

Ships after v0.3.0 (the Node port) is stable. Phase 3 line item from
the v0.3 rewrite plan: *"Localhost web UI — Fastify or Hono serving
the existing D3/Cytoscape templates, live graph data, recent recalls,
mass distribution, etc."*

The terminal view (`bellamem resume`) is the load-bearing product.
The web UI exists to make **things that are unreadable in text
readable in pixels** — edge neighborhoods, mass histograms, and
turn-by-turn trace replay.

---

## Differentiator: the session trace

Bellamem's advantage over claude-mem and other memory-for-agents
tools is **not** that it stores conversations. Everyone does that.
The advantage is that the graph stores *per-turn sources with mass
deltas and typed edges*. Every concept carries a list of source_refs
keyed by `{session_id}#{turn_idx}`, and every mass value on a concept
is the sigmoid of a sum of log-odds bumps, one per citation.

That data is already on disk. The terminal output flattens it to
ranked lists. The web UI should **restore the time dimension**:

- **Replay a session turn by turn** — watch concepts get created,
  cited, ratified, retracted as the conversation flows.
- **Concept birth vs ratification trace** — for any concept, show
  its mass accumulating over its source_refs in order, with voice
  crossings (user ↔ assistant) highlighted.
- **Cross-session panorama** — see which sessions birthed which
  invariants vs which ratified them later.

Claude-mem stores narrative summaries ("summary of session 42: we
decided X and fixed Y"). Bellamem stores a **graph with provenance**.
The web UI needs to show the provenance, not hide it behind a
D3 blob.

---

## Scope for v0.3.1 (one-week target)

**In scope:**
1. `bellamem serve` subcommand — Hono server on `localhost:7878`
2. `/api/*` JSON endpoints backed by the same `store.ts` the CLI uses
3. Three HTML views under `web/`:
   - **Overview** — audit signals + mass histogram + class × nature
     counts + recent decisions
   - **Graph** — D3 force-directed with class filter, session
     coloring, click-to-trace
   - **Trace** — session picker + turn-by-turn replay with live
     concept state and mass accumulation
4. SSE `/api/watch` — reload client when `.graph/v02.json` mtime changes
5. Read-only — no graph mutations from the UI

**Out of scope:**
- Auth, multi-user, https, anything beyond `localhost`
- Interactive ask/recall in the UI (v0.3.2)
- 3D force graph (v0.3.2 — Three.js template port)
- Graph editing, concept merging, manual retraction
- Running `save`/`ingest` from the UI

**Stop-ship gates:**
- Overview + Graph + Trace must all render against the live v0.2
  graph with no runtime errors
- SSE reload must fire within 200 ms of a file write
- Static bundle size < 200 KB (vendored D3 is the only big thing)

## Architecture

### Server

Single Hono app in `packages/bellamem/src/server.ts`. Reuses the
existing `store.ts` loader so the server and CLI share the exact
same graph-reading path. No server-side state — the graph is the
only state, and it lives on disk.

Why Hono over Fastify: Hono is ESM-native, 3x smaller, and has a
simple SSE middleware. For a localhost read-only tool, Fastify's
plugin ecosystem is wasted overhead.

```
GET  /                       → redirect to /overview
GET  /overview               → web/overview.html
GET  /graph                  → web/graph.html
GET  /trace                  → web/trace.html
GET  /static/*               → web/static/* (css, js, vendored d3)

GET  /api/graph              → full GraphJSON (same shape as v02.json)
GET  /api/resume             → resume_text as text/plain
GET  /api/audit              → audit report as JSON (signals + ephemeral)
GET  /api/sessions           → [{session_id, n_turns, first_ts, last_ts,
                                  n_concepts_touched, concept_samples}]
GET  /api/session/:sid       → { sources, concepts_touched, edges_from }
GET  /api/session/:sid/trace → turn-by-turn unroll:
                                [{turn, speaker, text_preview,
                                  cites, creates, edges}]
GET  /api/concept/:cid       → { concept, source_refs_expanded,
                                  incoming_edges, outgoing_edges,
                                  mass_history }
POST /api/ask                → { query, classFilter? } → ask_text JSON
GET  /api/watch              → SSE: {"event":"reload","mtime":...}
                               on .graph/v02.json mtime change
```

`mass_history` is synthesized on demand from the concept's
source_refs + the sequence of cite() calls that would reproduce the
current mass. Replaying from scratch gives the correct
per-turn-index mass value without needing to persist it.

### Client

Vanilla TypeScript compiled to a tiny bundle per page. No React,
no Vue. The static HTML shells import a per-page JS module:

```
web/
  overview.html        — <div id="app"></div> + <script src="/static/overview.js">
  graph.html           — <div id="app"></div> + vendored d3
  trace.html           — <div id="app"></div>
  static/
    app.css            — single stylesheet
    overview.ts        — fetch /api/audit + /api/graph, render bars + cards
    graph.ts           — port of Python viz_template_d3 to TS, live-reload
    trace.ts           — session picker + scrubber + concept panel
    d3.v7.min.js       — vendored, not pulled from CDN
```

Build with `esbuild` (single devDep, no bundler config fight). The
compiled JS lands in `packages/bellamem/web/static/` and is served
unchanged at runtime.

**No framework** because:
1. The whole UI is three pages with maybe 2,000 lines of client code
   total. React adds more kB than the app itself.
2. The Python viz templates are already vanilla D3. Porting them
   means preserving shape, not refactoring into components.
3. Fewer moving parts to hit version skew on during dogfood.

### SSE watcher

```ts
// server.ts
const watchers = new Set<ReadableStreamDefaultController>();
fs.watch(graphPath, { persistent: false }, () => {
  for (const c of watchers) c.enqueue(`data: {"event":"reload"}\n\n`);
});
```

Client side: `new EventSource("/api/watch")` + `window.location.reload()`
on the first `reload` event. Cheap, effective, 5 lines each side.

## Three views

### 1. Overview

Goal: answer "what's in this graph right now" in ≤ 3 seconds of
scanning.

Layout (single page, three stacked panels):
1. **Header** — `N concepts · M edges · K sources · X sessions`,
   plus the five audit signal badges (`concept_density`, `structural_edge_ratio`,
   `mass_floor_fraction`, `mass_spread`, `orphan_refs`) each color-
   coded ok/soft/hard.
2. **Mass histogram** — 10 buckets across [0.5, 1.0] as a bar chart.
   This is the `mass_spread` audit signal made visible. Click a
   bucket → jump to graph view filtered to that bucket.
3. **Class × nature grid** — 4×3 grid with concept counts per cell,
   each cell the top 3 concepts by mass. Clicking a cell opens the
   filtered graph view.
4. **Recent decisions** — last N decision concepts by
   `last_touched_at`, with linked turn source.

### 2. Graph view

Port of Python's `viz_template_d3.html` to TS. Key adaptations:
- **Node color** by class, saturation by mass.
- **Node size** by `source_refs.length` (log-scaled).
- **Edge style** per type: dispute = red dashed, retract = red
  dotted, cause = solid dark, support = thin gray, elaborate = blue.
- **Session coloring toggle** — when on, nodes are tinted by their
  `first_voiced_at` session, so you can see which sessions birthed
  which concepts. This is a claude-mem feature gap.
- **Filters** — class checkbox panel, nature checkbox panel, session
  selector, mass slider.
- **Click** a node → drawer with:
  - topic, class/nature, mass, voices
  - source_refs expanded to turn previews
  - incoming/outgoing edges with neighbor links
  - "show this concept in trace" button

This is the central view and the one that most of the D3 code lives
in. Budget 600-800 lines of client code.

### 3. Trace view — the differentiator

Goal: watch a session turn-by-turn as the graph updates, so the
provenance of every concept is visible.

Layout:
```
┌─────────────────────────────────────────────────────────────┐
│ session: [ dropdown ▼ ]   turn [ ◀ ] [ 42 / 567 ] [ ▶ ]    │
├─────────────────────────────────────┬───────────────────────┤
│                                     │  LIVE STATE at t=42   │
│  TURN  #42  [assistant]             │  ───────────────────  │
│  ───────────────────────            │  concepts touched     │
│  "We should ship v0.3 as a Node     │  in this turn:        │
│   package, not Python, so npx       │                       │
│   works from day one."              │  + walker primitive   │
│                                     │    m: 0.72 → 0.78     │
│  cites:                             │    (new voice: user)  │
│  - npm-vs-pypi (dispute)            │                       │
│  - node-ts-port-decision (support)  │  • concept map visual │
│                                     │    vocabulary         │
│  creates:                           │    m: 0.50 → 0.62     │
│  - npx install story                │    (new: created)     │
│    (invariant/normative)            │                       │
│                                     │  mini-histogram:      │
│  concept_edges:                     │  ┃ ┃ ┃ ▃ ▆ █ █ ▆ ▃   │
│  - node-ts-port-decision            │  0.5 ─────────── 1.0   │
│    → cause → npx-install-story      │                       │
└─────────────────────────────────────┴───────────────────────┘
  [scrubber: •────────•────▶────────●]  0%     50%    100%
```

Server contract: `GET /api/session/:sid/trace` returns the entire
turn-by-turn unroll as one JSON array. Client caches it and drives
the scrubber from memory — no round-trips per turn. Live state at
turn `t` is derived client-side by replaying the first `t` turns of
the trace through a JS reimplementation of `apply_classification`
(just the state tracking, not the persistence).

The scrubber sits in the URL as `?session=...&t=42` so traces are
shareable by link.

Click any concept name in the turn panel → open graph view filtered
to that concept's 1-hop neighborhood. Click a source_ref → open the
full turn text in a modal. Click "⟳" on a concept → play a mini-
animation of its mass history.

This is where bellamem earns its keep. Shipping this in v0.3.1 —
even in a rough form — makes the difference vs claude-mem legible.

## Serve command wiring

```
bellamem serve [--port 7878] [--graph PATH] [--no-open]
```

Default behavior:
1. Resolve graph path via the same `projectRoot` + `graphPathFor`
   logic as the rest of the CLI.
2. Bind `localhost:7878`. Bind is explicit — never `0.0.0.0`. If the
   port is taken, walk up `7879, 7880, ...` to `7900` then fail.
3. On startup, print:
   `bellamem web ui → http://localhost:7878  (graph: /abs/path/v02.json)`
4. Unless `--no-open`, open the default browser via `open`/`xdg-open`.
5. Watch the graph file; re-read on mtime change; push SSE reload.
6. Ctrl-C cleanly closes all SSE connections before exiting.

Logs are minimal: one line per incoming request (method + path +
status + ms), no framework noise. The server is a dev tool for one
user; production logging is not the shape.

## Dependency budget

New runtime deps:
- `hono` (~15 KB) — server
- `@hono/node-server` (~5 KB) — Node adapter
- `open` (~2 KB) — cross-platform browser launcher

New dev deps:
- `esbuild` (~10 MB installed) — bundles `web/static/*.ts`

**That's it.** No React, no Vite, no Tailwind, no data fetching
library, no charting library — D3 covers the charts we need and it's
already vendored for the graph view.

Install footprint target: npm install adds < 50 new packages.

## Risks

1. **Scope creep** — "add interactive ask" is the obvious next ask
   and will blow the week. Defer hard. Interactive ask is v0.3.2.
2. **D3 version skew** — the Python template uses D3 v7. Vendor the
   same version; don't let browser cache vs CDN mismatches bite.
3. **Trace unroll fidelity** — the client-side mass replay must
   produce the exact same numbers as the server-side
   `rebuildMassFromSourceRefs` path. Cover this with a cross-check
   test that asserts they agree for every concept in the live graph.
4. **Localhost-only trap** — some users will want to run bellamem
   remotely. Document the binding explicitly. Do not lift the
   localhost lock in v0.3.1; that's a separate security review.
5. **File watching on macOS** — `fs.watch` is unreliable on some
   macOS versions. If SSE reload flakes, fall back to a 2-second
   `fs.stat` poll. Cheap fallback, same client contract.
